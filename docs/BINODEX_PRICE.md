# Как правильно снимать цену binodex OTC (референс для других проектов)

Документ описывает проверенную на живом сайте схему «скриншот графика + цена под ним»
для binodex.app (OTC). Цель — чтобы цена в подписи **совпадала с ценой, нарисованной на
графике в кадре**. Схема портируемая: ниже суть, рекомендованный метод и его обоснование.

> **TL;DR.** Цена, нарисованная на ярлыке графика, доступна прямо на странице как
> **`window.chartData.price`** (`{symbol, price}`). Это и есть рекомендованный источник:
> берём **медиану нескольких быстрых чтений вокруг `screenshot()`** и округляем до decimals
> актива. WebSocket-котировки (§1) нужны для liveness/детекта отвала и как фолбэк — но как
> цену кадра WS брать не надо: **WS опережает график на ~150 мс** (см. §3).

## 1. WebSocket-фид котировок (источник данных графика, liveness, фолбэк)

На странице `/trade` цена нарисована внутри `<canvas>`. Числовой поток котировок идёт по
WebSocket — из него строится график, и он же нужен для детекта живости фида:

- **URL:** `wss://api-coins.binodex.io/market/?EIO=4&transport=websocket` (Socket.IO).
- **Событие (входящий фрейм):**
  ```
  42/graphic,["graphic",{"symbol":"EUR/USD-OTC","timestamp":1780307015001,"price":1.23065,"high":1.23066,"low":1.23065}]
  ```
  - `symbol` — `'<pair>-OTC'` (напр. `'EUR/USD-OTC'`);
  - `price` — котировка (5 знаков в потоке; график рисует округлённой — см. §5);
  - `timestamp` — серверное время тика (epoch ms), **не используем** (часы расходятся);
  - `high`/`low` — экстремумы строящейся свечи.
- **Частота:** ~5–6 тиков/с (тик каждые ~150–200 мс).
- **Подписка** (страница шлёт сама): `42/graphic,["graphic",{"method":"SUBSCRIBE","symbol":"EUR/USD","interval":"15s","otc":true}]`.

> Прочие WS (`api.binodex.io/market` → `pairs`, `api.binodex.io/binodex`) цену **не** стримят
> — слушать только `api-coins.binodex.io`.

Перехват ставить **до навигации** (поймать поток с самого старта):
```python
def on_ws(ws):
    if 'api-coins.binodex.io' in ws.url:
        ws.on('framereceived', lambda d: tracker.handle_message(getattr(d, 'payload', d)))
page.on('websocket', on_ws)
```

WS-трекер в этом проекте используется для: подтверждения, что выбранная пара прогрузилась
(пошли котировки), `feed_dead` (фид мёртв = WS закрыт и давно нет тика), и как **фолбэк**
цены кадра, если `chartData` недоступен.

## 2. Рендеренная цена — `window.chartData.price` (главный источник)

Движок графика держит текущую отрисованную цену прямо в глобале страницы:
```js
window.chartData // → { symbol: "EUR/USD-OTC", price: 0.6713642964343247 }
```
- `symbol` — `'<pair>-OTC'`, совпадает с WS-ключом; по нему проверяем, что цена от нужной
  пары (важно сразу после переключения, пока движок не перерисовал).
- `price` — **анимированное** значение, которое рисуется на ярлыке (много знаков; округление
  до decimals актива даёт ровно то число, что видно в кадре).

Проверка чтением: `await page.evaluate("() => window.chartData ? "
`"{symbol: window.chartData.symbol, price: window.chartData.price} : null")`.

## 3. Почему именно `chartData`, а не WS-тик (главное)

`chartData.price` = пиксели ярлыка (рисуются из него, синхронно — проверено сверкой кадров).
А **график отстаёт от WS-фида на ~150 мс**: свежий тик приходит по WS первым, и `chartData`
плавно (easing) доезжает до него за ~один интервал тика. Замер (кросс-корреляция
`chartData(t)` ↔ `WS(t − lag)`): максимум при **lag ≈ 150–160 мс, r ≈ 0.997**.

Следствие — типичная ошибка: брать цену из **WS** (даже «последний тик до кадра»). Тогда
подпись = свежая котировка, **которую график ещё не нарисовал**, и цена «убегает вперёд» от
картинки на ~150 мс. Сверка с нарисованным ярлыком (10 кадров):

| метод цены кадра | совпало с ярлыком |
|---|---|
| WS, последний тик до кадра | ~5/8 |
| `chartData`, одно чтение | ~5/8 |
| **`chartData`, медиана 3+3 чтений вокруг кадра** | **9/10** |

Одно чтение `chartData` не лучше WS (ярлык анимируется непрерывно — единичный сэмпл ловит
случайную фазу, иногда даёт анимационный выброс). **Медиана нескольких быстрых чтений вокруг
кадра** гасит выброс и даёт стабильное совпадение.

## 4. Правило снятия (рекомендованное)

1. `symbol = '<pair>-OTC'` для проверки принадлежности цены паре.
2. Несколько (≈3) быстрых чтений `window.chartData.price` **вплотную ДО** `screenshot()`.
3. `screenshot()` (момент кадра).
4. Ещё несколько (≈3) чтений **сразу ПОСЛЕ**.
5. Цена кадра = **медиана** всех чтений (с `chartData.symbol == symbol`). Если чтений нет
   (глобала нет / другая пара) — **фолбэк** на WS-цену по моменту кадра.
6. Округлить до **decimals актива** (§5) — как рисует график.

Остаточный промах ±1 в последнем знаке возможен только на резком тике (ярлык анимируется
непрерывно, мы сэмплируем дискретно) — это уже на грани читаемости знака, не лечится.

## 5. Decimals (число знаков) — per-asset

Число знаков у каждого актива своё и совпадает с тем, что рисует график. В этом проекте —
колонка `round` в `option_data.otc_data_view`. Эталон: EUR/USD=5, JPY-пары=2. В свою цену
надо подставлять то же число знаков, иначе подпись покажет лишний/недостающий знак.

## 6. Скелет кода (портируемый)

```python
import statistics

CHART_DATA_JS = ("() => { const c = window.chartData;"
                 " return (c && typeof c.price === 'number')"
                 " ? { symbol: c.symbol, price: c.price } : null; }")

async def read_chart_prices(page, symbol, count):
    out = []
    for _ in range(count):
        try:
            data = await page.evaluate(CHART_DATA_JS)
        except Exception:
            data = None
        if isinstance(data, dict) and (not symbol or data.get('symbol') == symbol):
            p = data.get('price')
            if isinstance(p, (int, float)):
                out.append(float(p))
    return out

# снятие кадра
symbol = f"{pair}-OTC"                          # 'EUR/USD' -> 'EUR/USD-OTC'
element = page.locator(chart_zone).first
await element.wait_for(state='visible', timeout=...)
reads = await read_chart_prices(page, symbol, 3)   # до кадра
await element.screenshot(path=shot_path)            # кадр
reads += await read_chart_prices(page, symbol, 3)  # после кадра
price = statistics.median(reads) if reads else ws_tracker.get_price_at(asset, t_shot)  # фолбэк
price = round(price, asset_decimals)                # как на графике
```

## 7. Чек-лист переноса в другой проект

- [ ] WS-перехват `api-coins.binodex.io` поставлен **до** `page.goto` (фид/liveness/фолбэк).
- [ ] Цена кадра берётся из `window.chartData.price`, **не** из WS (WS опережает график ~150 мс).
- [ ] Несколько чтений `chartData` ВОКРУГ `screenshot()` → **медиана** (гасит анимационный выброс).
- [ ] Проверять `chartData.symbol == '<pair>-OTC'` (не схватить цену чужой пары).
- [ ] Фолбэк на WS-цену, если `chartData` не отдал значений.
- [ ] Округление до per-asset decimals (совпадает с графиком).

## 8. История

Раньше цена кадра бралась из WS (последний тик до `t_shot`, с историей тиков и
синхронизацией по локальным часам). Это давало промах из-за лага отрисовки (~0–500 мс) —
WS опережал график. Обнаружено, что страница держит отрисованную цену в `window.chartData`;
переход на медиану её чтений закрыл этот промах. Диагностика — `scripts/probe_lag.py`
(замер лага WS↔chartData) и `scripts/probe_chartdata_median.py` (сверка медианы с кадром).
